# ADR 0001: サービスベースアーキテクチャを採用する

- Status: Accepted
- Date: 2026-06-17

## Context

『アーキテクチャの基礎』(Mark Richards & Neal Ford) のサービスベースアーキテクチャを
学習する題材として EC サイトを構築する。マイクロサービスのような細粒度・DB完全分離の
複雑さを最初から背負わず、「素直に立ち上げて改善で育てる」方針を取りたい。

## Decision

スタイルとして **サービスベースアーキテクチャ** を採用する。決め手は
「サービスが**粗粒度**（coarse-grained）であること」。ドメイン単位の 5 サービスに分割する。

- `product`（商品）、`order`（注文・横断の中心）、`payment`（決済）、`member`（会員）、`shipping`（配送）
- 各サービスは 1 コンテナ・個別デプロイ。内部は `internal/` で閉じる。
- UI もドメイン分割（storefront / mypage / admin）。出発点は UI → サービスの**直接呼び出し**。

改善ロードマップ:

- Step 0（現在）: 分割UI + 直接呼び出し + 共有DB（schema 分離）
- Step 1: API ファサードを足し UI 入口を集約（薄く保ち、段取りは各サービスに置く）
- Step 2: データ所有権を確定し schema 分離を徹底（他ドメインへの書き込みは持ち主経由）
- Step 3: 結合の弱い縁から DB を分割（payment / shipping → member → product × order）

## Consequences

- 粗粒度ゆえにサービス間通信が少なく、共有DBと組み合わせて初期の複雑さを抑えられる。
- 個別デプロイ可能だが、データは当面共有するため真の独立性は段階的に獲得する。
- マイクロサービスへ進化させる余地を残す（[[0005]] のフロント選定もこれを見据える）。

## Alternatives considered

- **モノリス**: 学習目的（サービス分割の体験）に合わない。
- **マイクロサービス**: 細粒度・DB分離・分散の複雑さを最初から負うことになり方針に反する。
