# ADR-202606261700: 在庫を増減レコードの台帳で持ち、予約→確定の2フェーズで引き当てる

- Status: Accepted
- Date: 2026-06-26
- Relates to: ADR-[[202606261214]] (冪等性。check-then-act を持たず DB で保証), ADR-[[202606231000]] (schema 所有権), ADR-[[202606261216]] (致命/縮退分類), ADR-[[202606190900]] (注文時スナップショット), GitHub #87

## Context

- product に在庫の概念が無く、checkout (`server/order/internal/handler/order_write.go`) は product を単価取得のためだけに呼ぶ。数量上限が無く売り越し (overselling) を原理的に防げない (#87)。
- カート→決済の間も在庫を確保したい。引当・確定・戻しを後から追える形にしたい。

## Decision

在庫を product schema 内に持ち、予約→確定の2フェーズで引き当てる。

- **単一の可変 stock 列を持たず、入庫/予約/確定/戻しを在庫変動レコード (台帳) として追記する**。利用可能在庫は `入庫 − 確定 − 有効な予約` で導く。在庫増減が監査可能になり、2フェーズの各操作が履歴として残る。
- **引当は予約→確定の2フェーズ**。checkout 受付で TTL 付き予約レコードを作り、決済確定で確定レコードへ昇格、キャンセル/期限切れで戻しレコードを積む。
- **超過販売は DB で強制する** (アプリ側 check-then-act を持たない。ADR-[[202606261214]] と同型)。利用可能在庫を下回る予約を原子的に弾く。
- **在庫不足は致命扱い** (ADR-[[202606261216]])。予約不能なら checkout を 409 で失敗させ、注文・決済を作らない。
- 在庫は商品と結合が強いため product schema 内に置く (ADR-[[202606231000]] の所有権方針と素直。コンテナも増やさない)。

## Consequences

- 売り越しが DB で防がれ、引当の履歴が残る。キャンセル時の在庫戻し (#88) が確定レコードの反対仕訳として自然に乗る。
- カート中も在庫を確保できる代わりに、予約 TTL の期限切れ回収 (遅延処理) が要る。
- 現在在庫が集計由来になるため、読み取りが重ければ残高のマテリアライズ (サマリ行) が将来要る。

## Alternatives considered

- **単一 stock 列の即時減算**: 実装は最小だが履歴が残らず、カート中の確保もできない。条件付き UPDATE で売り越しは防げるが 2フェーズ要件を満たさない。
- **別 inventory サービス/schema**: 在庫を独立量子にできるが単一事業者 storefront では過剰。増えてから移す。
- **TTL を持たない確定のみ引当**: カート段階の確保ができない。
