# ADR-202606261700: 在庫を append-only の在庫変動台帳で持ち、予約→確定の2フェーズで引き当てる

- Status: Accepted
- Date: 2026-06-26
- Relates to: ADR-[[202606261214]] (冪等性。check-then-act を持たず DB で保証), ADR-[[202606231000]] (schema 所有権), ADR-[[202606261216]] (致命/縮退分類), ADR-[[202606211200]] (payment.settled イベント), ADR-[[202606240522]] (DB-per-domain), GitHub #87

## Context

- product に在庫の概念が無く、checkout (`server/order/internal/handler/order_write.go`) は product を単価取得のためだけに呼ぶ。数量上限が無く売り越し (overselling) を原理的に防げない (#87)。
- カート→決済の間も在庫を確保したい。引当・確定・戻しを後から追え、在庫数を**値の上書きでなく増減の履歴から導ける**形にしたい。

## Decision

在庫を append-only の在庫変動台帳で持ち、予約→確定の2フェーズで引き当てる。

- **在庫変動台帳 (movement ledger) を単一情報源にする**。入庫 (+)、予約 (取り置き)、確定、解放/戻しを**すべて 1 行の変動レコードとして追記**する。**在庫数を保持するカラムを持たず、値の上書き (in-place mutation) を一切行わない**。現在の利用可能在庫は変動レコードの集計で算出する。
- **引当は予約→確定の2フェーズ**。checkout で TTL 付き予約レコードを追記 → 決済確定 (payment.settled を購読。ADR-[[202606211200]]) で確定レコードを追記 → キャンセル/期限切れで解放レコードを追記。
- **超過販売は DB で原子的に防ぐ** (アプリ側 check-then-act を持たない。ADR-[[202606261214]] と同型)。集計上の利用可能在庫を超える予約の追記を拒否する (具体手段は実装で詰める)。
- **在庫不足は致命扱い** (ADR-[[202606261216]])。予約不能なら checkout を 409 で失敗させ、注文・決済を作らない。
- **在庫は read-mostly な product カタログとは別の inventory schema に置く** (write-heavy で注文ライフサイクルに駆動される関心をカタログに混ぜない。ADR-[[202606231000]] の所有権を schema 単位で付与、ADR-[[202606240522]] の分割方向と整合)。

## Consequences

- 在庫増減が履歴として残り (監査可能)、予約・確定・戻しが反対仕訳として自然に乗る (キャンセル時の戻し #88)。
- カート中も在庫を確保できる代わりに、予約 TTL の期限切れ回収 (遅延処理) が要る。
- 現在在庫が毎回集計由来になる。読み取りが重くなれば集計の最適化 (マテリアライズドビュー等) を足すが、**変動台帳を単一情報源に保ち手で更新するカラムは作らない**。

## Alternatives considered

- **単一 stock 列の即時減算**: 実装は最小だが在庫を値で保持し上書き更新するため、履歴が残らず append-only 方針に反する。カート中の確保もできない。
- **product schema に在庫を置く**: read-mostly カタログに write-heavy な在庫を混ぜ、checkout のホットパスで product への書き込みを生む。ADR-[[202606190900]] が product を order の参照データと位置づけた前提とぶつかる。
- **新規 inventory サービス**: 在庫を独立量子にできるが単一事業者 storefront では過剰。まず schema 分離に留め、必要になれば切り出す。
