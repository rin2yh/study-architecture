# ADR-202606190900: 横断データは注文確定時に order 側へスナップショット保存する

- Status: Accepted
- Date: 2026-06-19
- Related: ADR-[[202606170909]] (顧客系/運用系 DB 分離・横断 JOIN 禁止), ADR-[[202606170903]] (横断データの将来論点), ADR-[[202606170901]] (codegen-first)

## Context

DB をドメインで物理分離し横断 JOIN を禁止した (ADR-[[202606170909]]) 一方、注文ドメインは他ドメインの属性 (商品名・単価) を本質的に必要とする。EC では「注文した時点の商品名・価格」が業務上の真実で、後から商品マスタが改名・改価しても確定済みの注文金額・明細表示は変わってはならない。横断データの選択肢は (a) 参照のたび持ち主へ問い合わせ (live reference) と (b) 必要時点で自ドメインへ複写 (snapshot) の 2 系統 (ADR-[[202606170903]])。

## Decision

横断データは **注文確定時 (checkout) に order 側へスナップショット複写** する ((b))。

- checkout 時に order が product を参照して権威ある属性を取得し、`order_items` に**値として複写**する。複写後は product の変更を遡及させない。合計金額もスナップショット単価から order 側で算出する。
- 「参照は確定の瞬間だけ・保持は値」とし、(a) の鮮度と (b) の不変性のいいとこ取りにする。確定後の照会は order 単独で完結し横断呼び出しが要らない。
- **複写するのは「確定時点で固定すべき業務事実」だけ** (商品名・単価・数量)。在庫や配送ステータスのように常に最新が欲しいものは複写しない。判断基準は「元が変わったとき過去の注文が追随すべきか」。

整合性の割り切り:

- order ヘッダ + 明細は **order 単独 DB の 1 ローカルトランザクションで強整合**。
- payment は別 DB・別サービスのため同 tx に含められず、**下流とは結果整合**にする。手配失敗は注文を残して `502` を返し、突合・再送は将来 Step に回す。分散トランザクション (2PC) や saga + 補償は Step 0 の規模に対し過剰なので入れない。
- **shipping は同期 checkout に含めない**。配送のトリガは決済確定であって注文確定ではなく (`pending` 決済で出荷枠を切ると未入金の出荷になる)、配送不調で checkout 全体を落とさないため、また配送は運用系の関心事 (ADR-[[202606170909]]) のため。決済確定後に運用系で手配する形にし、実装は後続イシューへ切り出す。

呼び出しクライアントは各サービスの OpenAPI から生成し (ADR-[[202606170901]])、handler は `internal/gateway` の出力ポート越しに使う (テストでスタブ可能)。

## Consequences

- **不変性**: 商品マスタの改名・改価が確定済み注文に影響しない。
- **読み取りが自律的**: 確定後の注文照会は order 単独 DB で完結する。
- **重複保持**: 商品名・単価が product と order に二重で載る。履歴保持の意図的な対価。
- **結果整合の宿題**: payment 手配の部分失敗は注文を残す。突合・再送 (または将来の saga 化) が要る。
- 生成クライアント (`internal/client/*`) は `/api`・`internal/db` 同様カバレッジ分母から除外する。

## Alternatives considered

- **(a) live reference のみ**: 過去注文の金額が現在値に揺れ、照会のたび横断呼び出しが走る。履歴の不変性に反する。
- **明細を UI 送信値に委ねる**: order が product を呼ばずに済むが価格をクライアントが詐称できる。権威は持ち主サービス側に置く。
- **分散トランザクション / saga + 補償**: 下流まで強整合にできるが Step 0 の規模に対しコスト過大。
