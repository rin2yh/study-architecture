# ADR 0008: 横断データは注文確定時に order 側へスナップショット保存する

- Status: Accepted
- Date: 2026-06-19
- Related: [[0004]] (schema 分離 / 横断データの将来論点), [[0012]] (顧客系/運用系 DB 分離・横断 JOIN 禁止), [[0013]] (migration 分割), [[0001]] (サービスベース)

## Context

[[0012]] で DB を顧客系 (order/payment/member) と運用系 (product/shipping) に物理分離し、
各サービスは **自身の所有 DB だけを向く / 横断 JOIN は禁止** とした。一方で注文ドメインは
本質的に他ドメインの属性を必要とする。たとえば注文明細には商品名と単価が要る。

横断データの扱いには 2 系統ある ([[0004]] の「将来の論点」):

- **(a) 参照のたびに持ち主サービスへ問い合わせる (live reference)**
- **(b) 必要時点で自ドメインへ複写して保持する (snapshot)**

EC の注文では「注文した時点の商品名・価格」が業務上の真実であり、後から商品マスタが
改名・改価しても **確定済みの注文金額や明細表示は変わってはならない**。(a) だけだと過去
注文の表示が product の現在値に引きずられ、履歴・会計・返金の整合が壊れる。

## Decision

横断データは **注文確定時 (checkout) に order 側へスナップショット複写** する ((b))。

- checkout の入力はカート (= `productId` + `quantity` の集合)。order は確定処理の中で
  **product を参照して権威ある属性 (商品名・単価) を取得** し、それを `order.order_items`
  に **値として複写** する。複写後は product の変更を遡及させない。
- 合計金額もスナップショットした単価から order 側で算出して `orders.total_cents` に持つ。
- 「参照は確定の瞬間だけ・保持は値」という形にすることで、(a) の鮮度と (b) の不変性の
  いいとこ取りをする。確定後の照会 (`GET /orders/{id}`) は order 単独で完結し、横断
  呼び出しは発生しない。

### スコープ: スナップショットするのは「確定時の事実」だけ

スナップショットは **確定時点で固定すべき業務事実** に限る (商品名・単価・数量)。在庫数や
配送ステータスのように「常に最新が欲しい」ものは複写せず、必要時に持ち主へ問い合わせる。
判断基準は「後から元が変わったとき、過去の注文がそれに追随すべきか」。追随すべきでない
ものだけ複写する。

### checkout のオーケストレーションと整合性

checkout は次の順で進める。

1. カート各行について product を参照し、商品名・単価をスナップショットして明細行を組む
   (product が無ければ確定不能 = 422)。
2. order ヘッダ + 明細を **1 ローカルトランザクション** で原子的に書く (横断 JOIN が無いので
   order 単独 DB の tx で閉じる)。
3. 確定した order に対して payment を **同期 HTTP で手配** する。

payment は別 DB・別サービスのため **手順 2 の tx には含められない**。ここでの失敗は注文を
残したまま `502 bad_gateway` を返す。すなわち本フローは

- **order 内 (ヘッダ+明細) は強整合**
- **下流 (payment) との間は結果整合** で、失敗時は要・整合復旧

という割り切りにする。分散トランザクション (2PC) や saga + 補償は導入しない (Step 0 の
規模に対して過剰)。復旧の作り込み (再送・突合) は将来 Step で扱う。

なお payment はここでは外部決済サービス (PSP) の薄い代役で、`status=pending` の記録までを
作る。オーソリ/キャプチャといった確定遷移は PSP 側の責務として本 Step では扱わない。

### shipping を同期 checkout に含めない

配送 (shipment) の作成は **checkout の同期パスに含めない**。理由:

- **決済確定が先**: 配送手配は決済が settle した後に始めるべきで、`pending` の決済の直後に
  出荷枠を切ると「未入金の出荷」になりうる。配送のトリガは決済確定であって注文確定ではない。
- **可用性を巻き込まない**: 注文・決済が成功しているのに配送サービスの不調で checkout 全体を
  `502` にするのは結合と失敗面を広げる。
- **ドメイン境界 ([[0012]])**: 配送は運用系 (ops) のフルフィルメント関心事。顧客系 checkout が
  同期で ops を駆動するより、決済確定を受けて ops 側が後追いで手配する形が素直。

したがって shipment は **決済確定後に運用系で手配** する想定とし、その実装 (イベント駆動 or
別エンドポイント) は後続イシューに切り出す。本 Step の order は shipping を呼ばない。

### 連携経路 (network)

[[0012]] の 4 network 構成上、order (external-private) から各サービスへの到達経路は次の通り。
order に env で base URL を注入する。

| 連携先 | 経路 | env (`*_API_URL`) |
| --- | --- | --- |
| payment | 同一 external-private に居るので直結 | `http://payment` |
| product | internal-private のため edge-proxy 経由 | `http://edge-proxy/product` |

呼び出しクライアントは各サービスの OpenAPI から oapi-codegen で生成する ([[0002]]
codegen-first)。生成物は `server/order/internal/client/<svc>/` に置き、handler はそれを
直接は触らず `internal/gateway` の出力ポート interface 越しに使う (テストでスタブ可能)。

### 実装 (order)

- `"order".order_items` を追加: `order_id` (FK), `product_id`, `product_name`,
  `unit_price_cents`, `quantity`。`product_name` / `unit_price_cents` が **スナップショット列**。
- `POST /checkout` を追加。リクエストは `memberId` / `paymentMethod` / `items[]`。

## Consequences

- **不変性**: 商品マスタの改名・改価が確定済み注文に影響しない。履歴・会計が安定する。
- **読み取りが速い・自律的**: 確定後の注文照会は order 単独 DB で完結し、横断呼び出しが要らない。
- **重複保持**: 商品名・単価が product と order に二重で載る。これは履歴保持の意図的な対価。
- **結果整合の宿題**: payment 手配の部分失敗は注文を残す。突合・再送の運用 (または将来の
  saga 化) が必要。known-issues ではなく本 ADR の既知の割り切りとして明示する。
- **配送は別トリガ**: shipment は決済確定後に運用系で手配する (後続イシュー)。order は本 Step
  では shipping を呼ばない。
- **カバレッジ**: 生成クライアント (`internal/client/*`) は `/api`・`internal/db` と同様、
  カバレッジ分母から除外する。

## Alternatives considered

- **(a) live reference のみ**: 確定後も毎回 product を引く。鮮度は最高だが過去注文の金額が
  現在値に揺れ、横断呼び出しが照会のたびに走る。本 ADR の目的 (履歴の不変性) に反する。
- **明細スナップショットをクライアント (UI) 送信値に委ねる**: order が product を呼ばずに
  済むが、価格をクライアントが詐称できる。権威は持ち主サービス側に置くべきで不採用。
- **分散トランザクション / saga + 補償**: 下流まで強整合にできるが、Step 0 の規模に対して
  実装・運用コストが過大。結果整合 + 復旧は将来に回す。
