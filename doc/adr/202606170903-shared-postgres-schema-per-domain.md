# ADR-202606170903: 共有 Postgres + ドメインごとの schema 分離

- Status: Accepted (「単一インスタンス」前提は ADR-[[202606170909]]、migration の「中央集約」前提は ADR-[[202606180900]] でそれぞれ更新。schema 分離方針自体は維持)
- Date: 2026-06-17

## Context

Step 0 は共有 DB から始める（ADR-[[202606170900]]）。一方で将来のデータ所有権確定・DB 分割
（Step 2/3）に備え、最初からドメイン境界を可視化しておきたい。マイグレーションと
sqlc のスキーマ入力を二重管理したくない。

## Decision

**共有 Postgres 1 つ**を、ドメインごとの **schema** で区画する。

- schema: `product` / `"order"` / `payment` / `member` / `shipping`
  （`order` は SQL 予約語のため、識別子としては常に二重引用符 `"order"` を用いる）
- マイグレーションは**中央集約**（`db/migration/`）で goose の連番ファイル（`00001_` 〜）。
  goose のバージョンテーブルは 1 ディレクトリ 1 つが前提のため、ディレクトリは分けず
  schema 修飾でドメインを分離する。
- sqlc は各サービスの `sqlc.yaml` で `schema: ../../db/migration` を入力にする
  （goose の `-- +goose Down` は無視される）。`omit_unused_structs: true` により
  各サービスは自分の query が触れる型だけを生成する。
- ファイル名は辞書順=数値順になるよう**ゼロ埋め必須**（sqlc は辞書順でパースするため）。

## Consequences

- マイグレーション（goose）と型生成（sqlc）でスキーマの単一情報源を共有できる。
- 各サービスの生成コードは自分のテーブルにスコープされる（`omit_unused_structs`）。
- Step 2 で「他ドメインへの書き込みは持ち主サービス経由」に寄せ、Step 3 で
  結合の弱い縁から DB（やがてマイグレーションディレクトリ）を分割していく。
- 横断データ（注文時の商品価格・名称など）は、DB 分割時に
  (a) サービス間参照 か (b) 注文時点のスナップショット保存 で扱う（将来の論点）。

## Alternatives considered

- **schema を分けず public に同居**: ドメイン境界が見えず、将来の分割が困難。
- **最初からドメインごとに DB を分割**: Step 0 の「素直に立ち上げる」方針に反する。
- **マイグレーションをサービスごとにディレクトリ分割**: goose のバージョンテーブルが
  ディレクトリ単位のため適用順序・横断管理が複雑。DB 分割時（Step 3）に移行する。
