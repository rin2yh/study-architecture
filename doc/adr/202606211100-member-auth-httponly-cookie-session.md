# ADR-202606211100: 認証は HttpOnly Cookie + member 所有のサーバ側セッション

- Status: Accepted
- Date: 2026-06-21

## Context

- 会員を認証したい。サービスベース構成 (ADR-[[202606170900]]) では認証の所有者を 1 つに定める必要があり、`password_hash`・セッションは「会員 identity」の一部なので member が持つのが自然。
- UI のデータ取得はサーバ側ローダに閉じる (ADR-[[202606170905]])。ブラウザは UI オリジンだけを叩く方針に合わせ、JWT を持たせる SPA 方式ではなくサーバ側セッションを採る。

## Decision

- 認証は member サービスが所有する (`member.sessions` と `members.password_hash` を追加)。パスワードは bcrypt ハッシュで保存。`password_hash DEFAULT ''` とし、空文字＝照合が必ず失敗する「ログイン不可の会員」を表す (既存行にダミーを撒かず移行できる)。
- セッションは不透明トークン (32 byte 乱数)。**SHA-256 を DB 主キーに保存し、生トークンは Cookie だけが持つ**。DB が流出しても生トークンを復元できず、既存セッションを乗っ取られないため。
- API は `POST /sessions` (ログイン) / `GET /sessions/{id}` (検証) / `DELETE /sessions/{id}` (ログアウト)。失敗・存在しないメール・パスワード不一致は文言を変えず 401 `unauthorized` (user enumeration を避ける)。エラーモデル (ADR-[[202606180901]]) に 401 を追加。ログアウトは存在しない id でも冪等に 204。
- セッションの `{id}` は string (不透明トークン) とし、int64 規約 (ADR-[[202606180903]]) から意図的に外す。連番だと総当たりでセッションを引けるため。
- UI (mypage / ADR-[[202606170908]]) は生トークンを **HttpOnly Cookie** (`member_session`) に載せ、サーバ側ローダだけが読む (XSS でのトークン奪取を防ぐ)。ローダはトークンで検証し、得た `memberId` を `X-Member-Id` で下流へ渡す。
- Cookie 属性は `HttpOnly; SameSite=Lax; Path=/` で **`Secure` は付けない**。ローカルスタックは edge-proxy が http 終端で、Secure Cookie は保存されずログインが成立しないため (TLS 終端を入れる段で付ける)。TTL は 7 日 (member 側と Cookie `Max-Age` を揃える)。

## Consequences

- 失効・ログアウトはサーバ側削除で即時に効く (ステートレス JWT と違いブラックリスト不要)。
- `X-Member-Id` は order が header パラメータとして受け取り、与えられた場合は本人の注文だけに絞る (未指定なら全件＝backoffice 等の内部用途は従来どおり)。
- ただし order は現状 `X-Member-Id` を**無検証で信頼**する。Step 0 は内部ネットワーク直呼び前提で許容するが、「誰がこのヘッダを付けてよいか」の信頼境界 (偽装で他人の注文を引ける問題) の確立は別 ADR に残す。それまでは認可ではなく「UI の利便のための絞り込み」と位置づける。
- **`Secure` 未設定は本番不適**。公開時は TLS 終端 edge を前提に有効化する。本決定は「ローカル完結・費用ゼロ」(ADR-[[202606170900]]) の制約下での妥協。
- path id の型が int64 / string の二系統になる。セッションだけの例外として固定する。

## Alternatives considered

- **ステートレス JWT をブラウザ保持**: サーバ側ストア不要だが失効が難しく、XSS でのトークン奪取に弱い。失効の挙動を素直に観察したい学習用途で不採用。
- **セッション id を連番 (int64) にして既存規約に合わせる**: 規約は揃うが推測可能になり総当たりで引ける。セキュリティ上不可。
- **生トークンをそのまま DB 主キーにする**: 単純だが DB 流出で全セッションを即乗っ取られる。ハッシュ保存で被害を限定する。
- **トークンを UUID にする**: v4 でも 256bit 乱数を低エントロピー化する利点が無く、v1/v7 は時刻・ノードを含み部分的に予測可能で秘密値に不適。標準形式・可読性は秘密のセッション id には無価値 (むしろログに出したくない)。UUID でも流出耐性のためハッシュ保存は必須で論点は独立。よって乱数トークン + ハッシュ保存を維持。
- **専用の auth/identity サービスを新設**: 認証を集約できるが、`password_hash`・セッションは member 集約の一部で、別サービスに割ると 1 集約が 2 サービスに分かれ突き合わせ＝サービス間結合・整合性問題を招く (集約分割はアンチパターン)。member が自分の認証を持つ方が方式に合い、Step 0 でのサービス追加は早すぎる複雑化 (ADR-[[202606170900]])。将来効くのは認証情報の保管ではなく検証・認可の強制点で、それは新サービスでなく edge-proxy / Step 1 の BFF へ寄せる筋。専用 identity が正当化されるのは SSO / OAuth / 外部 IdP / 複数認証元を束ねる段で、単一会員基盤の現状では当てはまらない。
