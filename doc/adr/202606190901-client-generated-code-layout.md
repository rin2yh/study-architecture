# ADR-202606190901: orval 生成物を gen/ ディレクトリに集約する

- Status: Accepted
- Date: 2026-06-19

## Context

`client/app/api/src/` 配下で orval 生成物と手書きコードが混在しており、生成物かどうかが
ファイル先頭の `Generated by orval … Do not edit manually.` ヘッダでしか判別できなかった。

- 手書き: `src/mutator.ts`, `src/mutator.test.ts`, `src/*.ts`（×5 バレル。`HTTPStatusCode*`
  の明示 re-export。ADR-[[202606180901]] のエラーモデル参照）
- 生成: `src/<service>/**`（サブディレクトリ配下すべて）

加えて、生成物を lint/format から除外する意図の `ignorePatterns`（`.oxlintrc.json` /
`.oxfmtrc.json`）が `package/api/src/*/**` という古いパスを指したまま壊れていた。実際の
パスは `package/` → `app/` のリネーム後 `app/api/src/...` であり、生成物が lint/format の
対象に入って `pnpm format` で orval 出力が再整形されていた。

## Decision

Go 側の規約（sqlc → `internal/db/*.sql.go`、oapi-codegen → `api.gen.go`、ADR-[[202606170901]]）と
同じく「生成物は専用の場所に揃える」方針を client 側にも適用する。

- orval の出力先を `src/gen/<service>/**` に集約する（`orval.config.ts` の `target` /
  `schemas` / zod `target`）。mutator は手書き `src/mutator.ts` のまま据え置き、生成側からの
  相対参照（`../../../mutator`）は再生成で自動調整される。
- 手書きは `src/` 直下に維持する（`mutator.ts`, `mutator.test.ts`, `*.ts` バレル）。
  バレルの import は `./<service>/...` → `./gen/<service>/...` に更新する。
- `package.json` の `exports` は手書きバレル（`./src/<service>.ts`）を指したままにする。
  公開境界はバレルであり、生成レイアウトの移動に追従させない。
- lint/format の `ignorePatterns` を実パス `app/api/src/gen/**` に修正する。
- ルート `.gitattributes` で `client/app/api/src/gen/**` を `linguist-generated=true` にし、
  GitHub の diff 畳み込みと言語統計からの除外を効かせる。

## Consequences

- 生成物と手書きがディレクトリ境界（`src/gen/` か否か）で一目で区別できる。
- ignore が実パスに一致したため、生成物は format 対象外になり orval ネイティブの整形
  （単一引用符等）がそのまま残る。手書き（バレル / mutator）のみ oxfmt の対象。
- 再生成は `pnpm --filter api gen`。出力先が `src/gen/` に固定され、手書きを巻き込まない。
- 生成物はコミットする方針（ADR-[[202606170901]] と同様、再現性のため）を維持する。

## Alternatives considered

- ヘッダコメントのみで区別を続ける → ignore がパス依存で壊れやすく、生成物が誤って整形
  される現状の問題が残る。ディレクトリ境界で根治する。
- バレルも `gen/` 配下に移す → バレルは手書き（重複 re-export の解消ロジックを含む）であり、
  生成物ではない。公開境界として `src/` 直下に残す。
