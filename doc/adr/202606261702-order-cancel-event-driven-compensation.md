# ADR-202606261702: 注文キャンセルの補償をイベント駆動 (order.cancelled) で各サービスに分散する

- Status: Accepted
- Date: 2026-06-26
- Relates to: ADR-[[202606211200]] (settled イベントで配送手配 = フォワード), ADR-[[202606261212]] (Outbox), ADR-[[202606250159]] (Redis span link), ADR-[[202606261214]] (冪等性), ADR-[[202606261700]] (在庫の予約・確定), GitHub #88

## Context

- フォワードフロー (order→payment→[settled]→shipping。ADR-[[202606211200]]) はあるが、逆方向 (返金・配送中止・在庫戻し) が一切無い。一度 checkout した注文を取り消せず、払った金額も戻せない (#88)。

## Decision

キャンセルをイベント駆動の補償トランザクションで分散する。

- **order がキャンセルを受け `order.cancelled` を発行**し、payment (返金) / shipping (未発送 shipment の中止) / 在庫の所有者 (予約・確定に対する戻しの変動レコードを追記。ADR-[[202606261700]]) が各自購読して自分の補償を行う。
- **フォワードの settled イベントと対称**にする。発行は Outbox (ADR-[[202606261212]])、トレースは span link (ADR-[[202606250159]]) で一貫させる。
- **各補償は冪等** (ADR-[[202606261214]])。再配信で二重返金・二重戻しにならない。
- **キャンセル可能境界は order が状態で判定**する (未発送=可、`shipped` 後=不可で返品フローに分岐)。

## Consequences

- 補償が各サービスに閉じ、order は下流の内部を知らない (疎結合)。
- 補償は結果整合で順次反映され、即時に全てが戻るわけではない。途中失敗は各サービスのリトライで収束させ、可視化する (ADR-[[202606261216]])。
- 発送済み注文の返品フローは本 ADR の対象外 (別 issue)。

## Alternatives considered

- **同期巻き戻し (order が payment/shipping を順に呼ぶ)**: フローは直接的だが order が下流に密結合し、途中失敗時の状態管理が order に集中する。フォワードが非同期イベントなのと非対称。
- **取り消し不可のまま運用**: EC として成立しない。
