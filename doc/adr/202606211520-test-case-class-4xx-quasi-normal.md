# ADR-202606211520: テストのケース分類で 4xx を準正常系・5xx を異常系に分ける

- Status: Accepted
- Date: 2026-06-21
- Relates to: ADR-[[202606180901]] (API エラーモデル) / ADR-[[202606180902]] (repository 実 DB 結合テスト) / [[test.md]] / [[go-test.md]]

## Context

テストのケース分類 (正常系 / 準正常系 / 異常系) は [[test.md]] に定義している。当初の `異常系` の例には
`404` やバリデーション失敗 (400) が混ざっていた。一方で実装は API エラーモデル (ADR-[[202606180901]]) で
クライアント起因の失敗を 4xx に正規化しており、stub 注入の error テスト (ADR-[[202606180902]]) でも
400 / 401 / 404 / 409 / 422 と 500 / 502 を同じ `異常系` グループに並べていた。

4xx と 5xx は性質が異なる:

- **4xx** (バリデーション失敗 400 / 未認証 401 / 未存在 404 / 競合 409 / 検証不能 422) は「入力や状態が
  不正」というクライアント起因の **想定済みの結果**。ハンドラが正しく弾けることを確認するケースで、
  起きること自体は設計の内側にある。
- **5xx** (DB エラーや上流障害に由来する 500 / 502) は **サーバが壊れて落ちる想定外**。

両者を `異常系` に同居させると、レビュー時に「想定済みのクライアントエラーを弾けているか」と
「サーバ障害時のフォールバックが妥当か」という別々の観点が混ざり、網羅の薄さに気づきにくい。
認証 PR (#18) のレビューでこの線引きが論点になり、スコープを分けて issue #28 として切り出した。

## Decision

**テストのケース分類で 4xx は準正常系、5xx (DB エラー等のサーバ起因の想定外) は異常系に置く。**

- **準正常系**: 想定内のエッジ・境界ケースに加え、クライアント起因の 4xx (400 / 401 / 404 / 409 / 422)。
- **異常系**: サーバ起因の想定外 (DB エラー / 5xx)。
- 判定の軸は HTTP ステータスではなく「誰起因の失敗か」。ステータスは目印にすぎず、例えば AppError で
  明示的に宣言した 502 のように「想定済みのサーバ応答」は準正常系に置いてよい。
- rdb 層の `ErrNotFound` (→404) / `ErrConflict` (→409) や gateway の上流 404 も、最終的に 4xx へ
  写るクライアント起因の結果なので準正常系に分類する。

既存テストの `name` 先頭ラベルも本分類へ一括で揃える ([[test.md]] の「新規追加・改修のタイミングで
揃える」を本 issue では一括適用)。

## Consequences

- **レビューの観点が分離する**: 「想定済みの 4xx を弾けているか」と「サーバ障害時のフォールバック」が
  別グループになり、それぞれの網羅の薄さに気づきやすい。
- **既存 ADR との整合**: ADR-[[202606180902]] は stub 注入による error テストを `異常系` と総称していたが、
  その `異常系` は「実 DB で再現しづらい error 経路を stub で注入する」という **検証手段の軸** であって、
  本 ADR の **ケース分類の軸** とは別物。当該 ADR は手段の記述として有効なまま残し、ケース分類の
  単一情報源は [[test.md]] + 本 ADR に置く (ADR-[[202606180902]] の Status 行に部分更新を注記)。
- **段階移行も可能**: 一括で揃えたが、以後の新規・改修でも同じ基準で分類すればよい。

## Alternatives considered

- **HTTP ステータスだけで機械的に振り分ける (4xx=準正常系 / 5xx=異常系 を例外なく適用)**: 単純だが、
  AppError で意図的に宣言した 502 のような「想定済みのサーバ応答」まで異常系へ落ちる。軸は
  ステータスではなく「誰起因か」に置き、ステータスは目印として使う。
- **分類を変えず `異常系` のままにする**: 4xx と 5xx が同居し続け、レビューで観点が混ざる元の問題が
  残る。test.md の例示を直すだけでは既存テスト名と乖離するため、名前も揃えた。
- **ADR-[[202606180902]] の本文を書き換えて旧表現を消す**: ADR は履歴として結論を消さない方針
  ([[adr.md]])。手段の軸とケース分類の軸は別物なので、当該 ADR は残し本 ADR で分類を上書きする。
