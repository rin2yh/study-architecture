# ADR-202606180903: 更新エンドポイント (PUT) はドメイン上ミュータブルな属性のみ置換する

- Status: Accepted
- Date: 2026-06-18
- Related: ADR-[[202606180901]] (API エラーモデル / 取得・作成), ADR-[[202606170901]] (codegen-first), ADR-[[202606170903]] (schema 分離)

## Context

ADR-[[202606180901]] で全サービスに取得・作成を足し、更新は後続とした (issue #4)。更新を足すにあたり 2 つの判断が要る:

1. **PUT で何を更新可能にするか** — 作成と同じ全項目を機械的に置換可能にすると、所有 FK・導出値・取引の事実・安定識別子まで可変になり、参照や会計の整合が壊れる。
2. **更新失敗のエラー正規化** — `UPDATE ... RETURNING` は no rows (404 相当) と unique_violation (409 相当) の両方が起こりうる。取得用 `FromRead` も作成用 `FromWrite` も片方しか拾えない。

## Decision

### 1. 更新可能なのは「ドメイン上ミュータブルな属性」だけ

`Update<Resource>Request` は id・生成時刻・所有 FK・導出値・取引の事実・安定識別子を除いた可変属性だけを持ち、PUT がそのサブセットを全置換する。order/payment/shipping は実質 `status` のみとなり、更新像が「状態機械の遷移」に収束する。金額・出荷情報など記録済みの事実は別操作 (返金・再出荷) で扱う。

検証は ADR-[[202606180901]] の規則を踏襲。更新で 422 が残るのは `product.priceCents` の負値チェックのみ。

### 2. 更新専用の正規化 `dberr.FromUpdate` を共通パッケージに足す

no rows → `ErrNotFound`、unique_violation → `ErrConflict` を両方扱う `FromUpdate` を追加し、全 repository が一様に使う。実際に 409 が起こりうるのは `member.email` だけだが、一様化で「更新は読み取り・書き込み双方で失敗しうる」性質を 1 箇所に閉じ込める。handler は到達しうるエラーだけ分岐する。

### 3. 動詞は PUT (全置換) のまま

可変サブセットを全項目必須で置換する冪等操作なので PUT。部分更新 (PATCH) は採らない (理由は Alternatives)。

## Consequences

- **不変条件が型と契約に出る**: 金額・取引手段・安定識別子・所有 FK が PUT body に無く、利用者も生成クライアント (orval) も「更新で書き換えられない」を契約として読める。ADR-[[202606170903]] とも整合する。
- **エラー対応が崩れない**: 404/409 が `FromUpdate` 1 関数に集約。不変列を更新しないので外部キー違反 (23503) は通常起きず、起きても 500 へ透過される。
- **将来の余地**: 状態遷移の業務ルール (例: `delivered` から `pending` へ戻せない) は handler に 422 として足せる。今回は素直な置換に留める。

## Alternatives considered

- **作成と同項目を全置換 (FK・金額含む)**: 実装は単純で対称的だが、所有関係の付け替えや事実の事後編集を許し、整合が壊れる。誤った API 像を与えるため見送り。
- **金額や出荷情報も更新可のまま (FK だけ不変)**: 当初案。導出値・取引の事実・安定識別子を可変にしてしまうため却下し、本 ADR で絞り直した。
- **PATCH で部分更新**: nullable/optional とゼロ値の区別が要り、薄い骨格には重い。全置換 PUT に倒す。
- **`FromUpdate` を作らず handler 側で個別判定**: db 依存 (pgx) が handler に漏れ、`dberr` に正規化を寄せた ADR-[[202606180901]] の方針に反する。
