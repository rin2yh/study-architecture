# コメント規約

コメントは **why（なぜそうするか・背景・非自明な制約）** だけを書く。コードを読めば分かること
（処理内容の言い換え）= **what コメントは書かない**。

- 対象は **全言語**（Go / TypeScript / SQL / YAML / shell など、go 以外も含む）。
- 意図は基本的に **命名（関数名・変数名・型）で語らせる**。名前で伝わるなら説明コメントは足さない。
- 例:
  - ✗ `// rows を id 昇順で詰め直す`（直後のコードそのまま = what）
  - ✓ `// pgxpool.New は遅延接続なので、不正な DSN でもここでは error にならない`（why）

理由: what コメントはコードが変わるたびに陳腐化し、嘘コメントの温床になる。why はコードからは
読み取れず、レビューや将来の変更時に最も効く情報なので、コメントの枠はそこに使う。

例外（残してよい）:

- 公開 API のドキュメントとして必要な doc コメント
- 生成コードの先頭マーカー（`// Code generated by ...`）等、ツールが要求するもの

## ADR で済む箇所は ADR の ID だけ参照する

判断の根拠が ADR に書いてある箇所は、コメントで根拠を**言い換えず ADR の ID 参照だけ**を残す
（`ADR-[[YYYYMMDDHHmm]]` 形式。ID は ADR のファイル名先頭のタイムスタンプ）。説明文を添えるのは
「その場でしか分からない why」がある場合に限る。

- ✗ `// 別 DB のため 1 tx に含められない。失敗は注文を残す（ADR-[[202606190900]] の Consequences）` ← ADR の要約 = 重複
- ✓ `// 下流の手配は決済確定後に分離 (ADR-[[202606190900]])` ← その場の判断 + 出典のみ
- ✓ 出典だけで足りるなら `(ADR-[[202606190900]])` の参照だけでよい

理由: ADR の内容をコメントに写すと二重管理になり、ADR を直したときにコメントが古くなる。根拠は
ADR を単一情報源にして、コードからはそこへ**ポインタ**を張るだけにする。
