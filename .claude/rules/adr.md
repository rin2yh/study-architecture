# ADR 作成規約

設計判断を記録するときの ADR の書き方をここに集める。CLAUDE.md の「設計判断は ADR で記載する」を
具体化したもの。新規 ADR は既存の `doc/adr/*.md` の体裁に合わせて書く。

## ADR を書く基準

CLAUDE.md の「設計判断は ADR で記載する」は、**すべての変更に ADR を起こす**意味ではない。次の
**2 条件を両方満たす判断だけ** ADR にする。片方でも欠けるなら ADR は作らず、必要なら根拠を
コード/PR 説明/`.claude/rules/` に残す。

1. **影響が局所で完結しない** — 複数サービス・量子をまたぐ、または公開 API / データ契約 /
   DB スキーマ / 認証・運用境界に触れる。
2. **覆すのにコストが高い、または実在する代替案の中から非自明なトレードオフを選んだ** —
   後から戻すのが大変、もしくは「なぜ採らなかったか」を残す価値のある対案があった。

書かなくてよい例 (どちらかの条件を欠く):

- 機械的・局所的なリファクタ (命名・抽出・整形)
- 容易に元へ戻せるツール / CI 設定の調整 (例: CI ステップを composite action へ括り出す)
- 実質的に代替案を議論していない自明な変更

理由: 粒度の細かい判断まで ADR にすると一覧が埋もれ、本当に重要な設計判断 (量子分割・DB 分離・
認証方式など既存 ADR 相当) が探しにくくなる。ADR は「後から経緯を知りたくなる重い判断」に枠を使う。

迷ったら ADR にせず、判断を仰ぐ (CLAUDE.md の ask 方針)。

## 1 ADR には決定を 1 つだけ

ADR にする判断 (上記 2 条件を満たすもの) が複数あるなら、**1 ADR = 1 決定**にする。1 つの ADR に
複数の決定を詰めない。複数あるなら ADR を分割し、相互は `Relates to` で参照する。

- 例: 「可観測性スタックの選定」と「非同期イベントのトレース伝播方式」は別の決定 → 別 ADR にする。
- 見分け方: その節だけ後から単独で覆せるか。単独で `Superseded` にしたくなる粒度なら、別 ADR に切る。
- 共通の背景は各 ADR の Context に簡潔に書き、重複は要約せず参照 (`ADR-[[<ID>]]`) で張る。

理由: 1 ADR に複数決定を混ぜると、片方だけ覆したいときに Status・本文の管理が破綻し、一覧から
「どの決定がどこにあるか」を辿れなくなる。決定ごとに ID を振れば、Supersede も相互参照も決定単位で効く。

注意: これは「細粒度の自明な変更まで ADR にする」という意味ではない (上記基準を欠くものは ADR に
しない)。あくまで **ADR 化すると決めた判断が複数あるときの、分割の粒度**を定めるもの。

## 置き場所とファイル名

- 置き場所は `doc/adr/`。
- ファイル名は `YYYYMMDDHHmm-slug.md`。先頭は **作成日時タイムスタンプ (分粒度)** で、これが ADR の ID。
  採番規約の根拠は ADR-[[202606211000]]。
- `slug` は内容が分かる英小文字ハイフン区切り (例: `cross-domain-snapshot`)。

## 本文の構成

1 行目を見出し、続けてメタ情報、その後に本文セクションを置く。

- 見出し: `# ADR-<timestamp>: <タイトル>` (例: `# ADR-202606170900: サービスベースアーキテクチャを採用する`)
- メタ情報 (箇条書き):
  - `Status:` — `Proposed` / `Accepted` / `Superseded`。全面的に後継へ移ったら Status 行に
    インラインで `Superseded by ADR-[[<ID>]]` と後継を書く。下記ライフサイクル参照。
  - `Date:` — `YYYY-MM-DD`。
  - 必要なときだけ標準メタ行を足す:
    - `Supersedes: ADR-[[<ID>]]` — この ADR が置き換える対象 (置き換えた新 ADR 側に書く)。
      範囲を限定するなら ID の後ろに対象部分を添える。
    - `Relates to:` — 関連する ADR・ファイルへの参照。
- 本文セクション (この順序・見出しで統一):
  - `## Context` — なぜこの判断が要るか。背景・制約・問題。可能なら推測でなく計測・実例で語る。
  - `## Decision` — 何を決めたか。決め手 (why) を明示する。
  - `## Consequences` — 採用した結果の良し悪し。トレードオフや今後の含みも書く。
  - `## Alternatives considered` — 検討して**採らなかった**案と、不採用の理由。

**簡潔に書く**。各セクションは要点を短い箇条書きで。why に絞り、コードや他 ADR から読めることの
言い換え・長い散文・同じ理由の繰り返しは載せない。冗長な ADR は要点が埋もれて読まれない。

## Status のライフサイクル

- 新規は `Accepted` で起こす。理由: レビューで問題があれば指摘される運用のため、`Proposed` で
  寝かせず確定として進める。見送り・差し戻しになったものだけ後から Status を落とす。
- 既存判断を**全面的に**置き換える ADR を起こしたら、新 ADR に `Supersedes: ADR-[[旧ID]]` を足し、
  旧 ADR の Status 行を `Superseded by ADR-[[新ID]]` に変える (双方向リンク)。
- **一部だけ**更新する場合は旧 ADR を `Accepted` のまま残し、Status 行の括弧で「どの部分が
  ADR-[[新ID]] で更新 / Superseded されたか」を添える (例: ADR-[[202606170901]])。
- いずれも古い ADR の本文や結論は消さない (履歴として残す)。

## 相互参照

- ADR どうし・コード中コメントからの参照は `ADR-[[<ID>]]` 形式 (例: `ADR-[[202606190900]]`)。
  ID は参照先ファイル名先頭のタイムスタンプ。
- 既存判断を要約で繰り返さず、参照だけ張る (重複管理を避ける。コメントからの参照方針は [[comments.md]])。

## 作成後にやること

- `doc/adr/README.md` の一覧表に **1 行追記** する (ID リンク・Status・タイトル)。ID 昇順を保つ。
- 既存 ADR を `Superseded` にした場合は、その行の Status も更新する。
- **書き上げた ADR を `simplify-adr` スキルでセルフ点検する**。上の「簡潔に書く」に照らし、コード/表の転記・他 ADR の要約再掲・同じ理由の繰り返し・what 散文が残っていないか確認し、あれば畳む (決定と why は残す)。起案直後に通すことで、後から一覧をまとめて直す手戻りを防ぐ。

理由: 体裁・セクション・ID 採番・一覧の更新を揃えておくと、ADR が増えても一覧から辿りやすく、
相互参照が壊れない。判断の根拠は ADR 本文を単一情報源にし、他所からは ID 参照でポインタを張る。
