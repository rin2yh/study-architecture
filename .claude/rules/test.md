# テスト規約

新規追加するテストは原則テーブル駆動で書く。

- **Go**: `tests := []struct{ name string; ... }{...}` を作り `for _, tt := range tests { t.Run(tt.name, func(t *testing.T) { ... }) }` で回す
- **TS (vitest)**: 同じ shape を持つケースなら `it.each([...])` を使う

理由: 同種のケースを一覧で見せた方が抜けが分かりやすく、ケース追加も「行を増やすだけ」になる。

ただし次の場合はテーブル化を**しない**:

- テーブル化のために本体に `if` / `switch` や追加ヘルパー関数を増やすことになる
- ケースが 1 つだけ、あるいは shape が大きく違う異質ケースだけ
- 既存の素直な `Test...` / `it(...)` を **テーブル化のためだけ** に書き直す (新規追加でテーブルにする方が安い)

可読性とテーブル化の天秤で、テーブル化が「行が増えるだけ」で済まないなら個別テストで書く。

## ケース分類

新規テストは「正常系 / 準正常系 / 異常系」で**明示的にグルーピング**する。理由は、レビュー時に
「Happy path しか書いてない」「異常系の網羅が薄い」を一目で気づけるようにするため。

- **正常系**: 期待通りの入力に対する期待通りの結果 (Happy path)
- **準正常系**: 想定内のエッジ・境界ケース (空配列、最大値、未ログイン、未指定など)
- **異常系**: 期待しないが起こりうる失敗 (DB エラー、404、500、バリデーション失敗など)

書き方の例:

- **Go**: 親 `t.Run` で分類する形 (`t.Run("正常系", func(t *testing.T) { ... })`)、あるいは
  テーブル駆動なら struct に `category string` を持たせて `t.Run(tt.category+"/"+tt.name, ...)`
  にする
- **TS (vitest)**: `describe` のネスト (`describe("正常系", () => { it(...) })`)、あるいは
  テーブル駆動の `it.each` を 3 つに分けて記述

「該当ケースが 0 個」 (例: 異常系を書きようがない pure 関数) なら省略してよい。

既存テストの強制リライトは不要 — 新規追加・改修のタイミングでこの分類に揃える。
