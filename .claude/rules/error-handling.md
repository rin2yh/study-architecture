# エラーハンドリング規約

エラーを **noop で握り潰さない**。エラーを明示的に捕まえる（TS の `catch`、Go の `if err != nil`）
のは、次の「意味のある」処理をするときだけにする。

- **意味のあるフォールバックにする**: 失敗を想定済みの値へ変換する（例: セッション検証失敗 →
  `return null` で未ログイン扱い、bind 失敗 → 400 を積む）
- **ログを出す**: その場で握って続行するなら最低限ログに残す（TS は `console.warn` 等、Go は
  `slog`）。後段の挙動が変わらなくても、失敗が起きた事実は可視化する

上のどちらもしない＝**何もすることがないなら、そもそも捕まえない**。

- TS: `throw` は自動で上位へ伝播する。「`catch` して noop」も「`catch` して素の `throw e`
  （再スロー）」も無駄なので書かない。捕まえず素通しすればよい（再スローは `no-useless-catch`
  でも弾かれる）。
- Go: `_ = err` でエラーを捨てない。ここで扱えないなら `return err` で素直に伝播する（これは
  無駄な再スローではなく、Go の正規の伝播）。

理由: 握り潰すと失敗が不可視になり、デバッグ不能・「嘘の成功」の温床になる。`catch {}` /
`catch { /* noop */ }` / `_ = err` での無言の破棄は、後から「なぜ動かないのか分からない」を生む。

- ✗ `try { await deleteSession(t); } catch { /* noop */ }`（失敗が消える）
- ✗ `try { ... } catch (e) { throw e; }`（素の再スローは無駄。捕まえず素通しでよい）
- ✓ `try { await deleteSession(t); } catch (e) { console.warn("...", e); }`（続行するが可視化）
