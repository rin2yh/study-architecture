# ADR Simplifier

`doc/adr/*.md` の既存 ADR を `.claude/rules/adr.md` の「簡潔に書く」基準へ retrofit するスキル。`rule-creator` がルールの新規作成を担うのに対し、本スキルは**既に書かれた冗長な ADR を、判断を変えずに表現だけ整理する**ワークフロー。

冗長な ADR は要点が埋もれて読まれない (ルール本文の理由)。新規 ADR だけ簡潔でも、一覧の大半が冗長なままだと「ADR は読むもの」という前提が崩れる。本スキルは既存分を同じ基準にそろえる (issue #65)。

## いつ使うか

- **新規 ADR を書き終えた直後**に、その 1 本が「簡潔に書く」基準を満たすかセルフ点検したいとき (冗長なら同じ基準で畳む)
- 既存 ADR が冗長で、決定と why に絞って短くしたいとき
- 「簡潔に書く」基準の制定前に書かれた古い ADR を、一覧の他と同じ密度にそろえたいとき
- テーマ単位 (API エラー系 / 可観測性系 / フロントエンド系など) でまとめて棚卸ししたいとき

新規 ADR を**書く・採番する作業そのもの**ではない (起案は `.claude/rules/adr.md` に従って直接書く)。本スキルが担うのは書き上げた後の簡潔さ点検・整理。設計判断の見直し (Supersede) でもない — 内容を変える必要が出たら別 ADR / 別 issue にする。

新規 ADR 直後の点検では、対象はいま書いた 1 本だけ。棚卸しの手順 (下記 1.) は飛ばし、その 1 ファイルに「何を削るか」を当てる。

## 大前提: 判断は変えない

これは**表現の整理であって設計判断の見直しではない**。次を必ず守る。

- **決定 (Decision) とその決め手 (why) は落とさない**。何を・なぜ選んだかは ADR の核。
- **結論・Status・採らなかった案 (Alternatives) は消さない**。古い判断も履歴として残す (ルールの Status ライフサイクル)。
- ID・ファイル名・タイムスタンプ・README 一覧の行は変えない (採番は ADR-[[202606211000]])。
- 迷う削除 (これは why か what か、消すと判断が辿れなくなるか) は CLAUDE.md の ask 方針で判断を仰ぐ。

## 何を削るか (冗長パターン)

`.claude/rules/adr.md`「簡潔に書く」と [[comments.md]] と同趣旨。次を削る:

1. **コード/YAML/表の転記** — 実装・スキーマ・compose から読めることの写し。ソースが SSOT のものは ADR に写さず参照だけにする (例: `code: string` の例示、struct タグ一覧、compose のポート/network 表)。
2. **他 ADR の結論の要約再掲** — `ADR-[[<ID>]]` 参照に置き換える。要約で繰り返さない (相互参照ルール)。
3. **同じ理由の繰り返し** — Context / Decision / Consequences に同じ why が三回出るなら一度にする。
4. **what の散文** — 処理内容の言い換え。why に絞り、コードから読めることは書かない。
5. **過剰なファイルパス・手順の列挙** — 「どのファイルに何を置くか」の網羅はコード/PR が SSOT。判断に必要な分だけ残す。

逆に**残す**: 決定、決め手 (why)、トレードオフ、不採用案とその理由、その ADR でしか分からない制約・経緯。

## 手順

### 1. 対象を洗い出す

- `doc/adr/*.md` (README を除く) を棚卸しする。`wc -l doc/adr/*.md | sort -rn` で行数の多い順に当たりをつける (冗長さの代理指標。長い ≠ 必ず冗長だが、転記・繰り返しは長い ADR に出やすい)。
- ユーザーがテーマやファイルを指定したらそれを優先。「残り全部」なら既に簡潔なものを除いて全件。
- 1 回で全部直すより、関連する塊 (テーマ) ごとに進めるとレビューしやすい (issue #65 の進め方)。

### 2. 1 ADR ずつ整理する

各 ADR について:

1. **Read** して決定・why・不採用案を把握する。
2. 上の「何を削るか」に照らして冗長箇所を特定する。**決定と why は必ず保持**。
3. 本文の構成 (`Context` / `Decision` / `Consequences` / `Alternatives considered`) と見出し・メタ行・ID は `.claude/rules/adr.md` どおり維持したまま、各セクションを短い箇条書きへ畳む。
4. 削った結果、参照すべき SSOT (ソース・他 ADR) への `ADR-[[<ID>]]` ポインタが張れているか確認する。要約を消したなら参照を残す。
5. **Write / Edit** で書き直す。

### 3. 検証する

- `git diff --stat doc/adr/` で削減を確認する (insertions ≪ deletions が期待形)。
- 決定文・Status 行・各 Alternatives が残っているか自分の差分で確認する (消えていたら戻す)。
- 相互参照の `ADR-[[<ID>]]` が実在ファイルを指すか確認する (`ls doc/adr/` で ID を突合)。削る前から張られていた参照を壊さない。
- README 一覧 (ID / Status / タイトル) は据え置き。書き直しで Status を動かさない。
- 直したファイルパスをユーザーに伝え、必要なら作業ブランチへ commit/push する。

## やってはいけないこと

- 決定・why・不採用案・結論を削って意味を痩せさせる (短さのために判断を失う)。
- Status を勝手に変える / Supersede を足す (表現整理は判断変更ではない)。
- ID・ファイル名・タイムスタンプ・README 行をいじる。
- 既存の `ADR-[[<ID>]]` 参照を壊す、または要約を消したのに参照を張り忘れる。
- 1 PR に全 ADR を詰め込んでレビュー不能にする (テーマ単位に分ける)。
