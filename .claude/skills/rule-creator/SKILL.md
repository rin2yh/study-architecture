# Rule Creator

`.claude/rules/` 配下のプロジェクトルールを作成・編集するスキル。`skill-creator` のルール版にあたる。

ルールは「Claude がこのリポジトリで作業するときに守るべき約束事」を Markdown で書いたもの。CLAUDE.md と違い、**特定のファイル種別やディレクトリに閉じた指示**を、必要なときだけコンテキストに読み込ませられるのが利点。

## いつ使うか

- 会話の中で「今後はこうする」と決めた方針を恒久化したいとき
- 特定のファイル（例: `*.test.ts`、`src/api/**`）にだけ効かせたい規約があるとき
- CLAUDE.md が肥大化してきて、トピック別・パス別に切り出したいとき

逐次手順や「呼び出したときだけ動く」ワークフローは**ルールではなくスキル**（`skill-creator`）が適切。常に全セッションで必要な普遍的な指示は CLAUDE.md が適切。迷ったら下表で判断する。

| 永続化したいもの | 置き場所 |
|---|---|
| 全セッションで常に効く普遍ルール | `CLAUDE.md` |
| トピック別・パス別の規約 | `.claude/rules/*.md` |
| 呼び出し時だけ実行する手順 | スキル（`.claude/skills/`） |
| 機械的に強制したい禁止事項 | PreToolUse フック |

## 公式仕様（`.claude/rules/`）

迷ったら公式ドキュメントで最新仕様を確認する: https://code.claude.com/docs/en/memory （"Organize rules with `.claude/rules/`" 節）。記載時点の要点は以下。

- 置き場所は `.claude/rules/*.md`。サブディレクトリも再帰的に読み込まれる（`frontend/` 等で整理可）
- 1ファイル1トピック。ファイル名は内容が分かるもの（`testing.md`、`api-design.md` 等）
- **パススコープ**はフロントマターの `paths:` キー（YAML 配列）で指定する
  - `paths:` がある → マッチするファイルを Claude が読むときだけコンテキストに載る
  - `paths:` がない → 全セッションで常時読み込まれる（`.claude/CLAUDE.md` と同等の優先度）
- glob 例: `**/*.ts`（全 TS）、`src/**/*`（src 配下全部）、`*.md`（ルート直下の md）、ブレース展開 `src/**/*.{ts,tsx}`

パススコープ付きルールの最小形:

```markdown
---
paths:
  - "**/*.test.ts"
---

# テスト規約

- （ここに規約を箇条書きで書く）
```

## 手順

### 1. 意図を捉える

ユーザー（または直近の会話）から次を引き出す。会話に既に答えがあるなら、推測で埋めず確認を取る。

1. **何の規約か** — どんな振る舞い・書き方を統一したいか
2. **なぜ必要か** — 背景・理由（ルール本文に「理由」として残す価値があるか判断する）
3. **対象範囲** — 全ファイルか、特定のファイル種別／ディレクトリか
4. **既存ルールとの関係** — 既存ファイルへの追記か、新規ファイルか。矛盾する規約がないか

### 2. スコープとファイルを決める

- 既存の `.claude/rules/` を `ls` で確認し、同テーマのファイルがあれば**追記**を優先する（規約の分散を防ぐ）
- 新規なら内容が分かるファイル名にする（例: `test.md`、`naming.md`、`api.md`）
- パススコープが必要なら、リポジトリの実ファイルを `Glob` で確認して **過不足のない glob** を決める
  - 例: 「TS のテストファイル」なら実在拡張子を確認（`.test.ts` だけか `.test.tsx` もあるか）してから `paths` を書く

### 3. ルールを書く

このリポジトリの既存ルール（`.claude/rules/*.md`）の作法に合わせる:

- 見出しは `# 〇〇規約`
- 箇条書きで簡潔に。**why（なぜそうするか）を書く**。what はコードから読めるので冗長に書かない
- 重要な規約は理由・代替案をネストして添える（既存の `architecture.md` が好例）
- 文体は**日本語かつ適度な敬語**（CLAUDE.md の開発ルールに準拠）
- 1ファイルが長くなりすぎないようにする（肥大化したらトピックで分割）

過度な `MUST`／`NEVER` の羅列は避け、理由を説明して Claude が納得して従える形にする。

### 4. 検証する

- フロントマターが正しい YAML か（`paths:` は配列）。`paths` 不要なら付けない
- glob が対象ファイルに実際にマッチするか `Glob` で確認する
- 既存ルール・CLAUDE.md と矛盾しないか見直す
- 書いたファイルパスをユーザーに伝え、必要なら commit/push する（このリポジトリでは作業ブランチへ）

## 例

会話で「分岐や関数が増えない範囲でテーブル駆動テストにしたい」と決まり、TS テストファイルに限定するケース:

`.claude/rules/test.md`

```markdown
---
paths:
  - "**/*.test.ts"
---

# テスト規約

- 分岐や関数が増えない範囲で、なるべくテーブル駆動テスト（`it.each` 等）にする
  - 理由: 同種のケースを表形式で一覧でき、ケース追加が容易で網羅性を読み取りやすい
  - 注意: テーブル化のために条件分岐やケースごとのコールバック関数が増えるなら、可読性を優先して個別のテストに分ける
```

## やってはいけないこと

- 逐次手順や「実行する作業」をルールに書く（それはスキルの領分）
- 対象を確認せずに当て推量の glob を書く
- 既存ルールと重複・矛盾する規約を別ファイルに増やす
- why を書かず what だけを並べる、あるいは理由なき MUST を羅列する
